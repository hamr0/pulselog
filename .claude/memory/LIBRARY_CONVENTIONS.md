# Building & publishing a JS library — standard advisory

The fixed reference for how I build a publishable JavaScript library. Its spine
is the **types convention** (JSDoc → `.d.ts` → CI, with no drift); around that it
fixes the library's shape, its adopter-facing `context.md`, and its doc set.

**Start with [`AGENT_RULES.md`](AGENT_RULES.md).** That is the parent standard
(POC-first, dependency hierarchy, simple-over-clever, open-source-only, security
invariants, the testing trophy). This advisory does **not** repeat it — it adds
only what is specific to shipping a JS *library* to npm. When the two ever seem
to disagree, AGENT_RULES wins.

---

## 1. Shape of a published JS library

- **Pure ESM JS + JSDoc. No build step for shipped code.** The `.js` you author
  is the `.js` that ships. No transpile, no bundler. (TypeScript appears only as
  a dev-only types toolchain — §2.)
- **One production dependency is the bar**, not a target. Stdlib-first per the
  AGENT_RULES dependency hierarchy; a second prod dep needs explicit
  justification. `typescript` / `@types/node` are **devDependencies** — they
  never ship and don't count against this.
- **`package.json` essentials:** `"type": "module"`; `main`; `exports` with a
  `types` condition on **every** subpath; `files` ships `src/` + `types/` + the
  doc set; `engines` pins the Node floor; `bin` if there's a CLI.
- **Config from the environment** (12-factor), never hardcoded; a value-less
  `.env.example` / `config.example.*` is the only committed config.
- **Scope discipline — mechanism in the lib, policy in the adopter.** Default
  *out* on new API surface; the burden of proof is on adding. Curation,
  thresholds, branding, cron, override lists belong to the adopter. Once a lib
  hits 1.0, only security fixes, bug fixes that don't change the API,
  doc corrections, and pulling-existing-mechanism-into-a-helper ship after.

## 2. Types: JSDoc → `.d.ts`, with no drift (the core)

### Why two artifacts, two audiences

| | JSDoc (`/** */` in source) | `.d.ts` (generated) |
|---|---|---|
| Serves | whoever **reads/edits the source** | whoever **`npm install`s the package** |
| Lives | inline, next to the code | shipped in the tarball, read by the consumer's editor |
| Crosses the npm boundary? | **no** — consumers' tooling ignores `.js` in `node_modules` | **yes** — the only thing that gives adopters autocomplete + type errors |

JSDoc helps contributors; the `.d.ts` makes the package pleasant to *use*. The
`.d.ts` is **generated from the JSDoc** so the two can't disagree.

### The anti-drift contract

No drift between JSDoc, `.d.ts`, and CI **iff all five hold**:

1. **JSDoc is the only hand-authored source of types.** No hand-edited `.d.ts`
   (lone exception: a minimal ambient shim for an untyped dep whose types you
   actually re-export — recipe step 7 below).
2. **`.d.ts` are generated by `tsc`, git-ignored, and built on publish**
   (`"prepublishOnly": "npm run build:types"`). Never committed → no stored
   artifact can go stale → drift is structurally impossible, not just discouraged.
3. **`tsc` runs `checkJs` + `strictNullChecks`, NOT full `strict`.** Full strict
   is ~95% `noImplicitAny` annotation-noise over framework-agnostic params and
   ~5% real null-safety; `strictNullChecks` keeps the 5% and drops the noise.
4. **CI runs `tsc --noEmit` on every push/PR _and_ in publish before publishing.**
   A JSDoc/code mismatch is a type error → it blocks merge and blocks publish.
   This is the gate that turns the JSDoc into a checked contract.
5. **`exports.types` (+ top-level `types`) point at the generated `.d.ts`, and
   `files` ships them.** Every `exports` subpath gets its own `types` condition.

Layout (where `.d.ts` land, build-hook name) does **not** affect drift — #2 + #4
do. Recommended default: emit to `./types` (keeps `src/` free of build output).

### Recipe

1. `npm install -D typescript @types/node` (dev-only).
2. `tsconfig.json`:
   ```jsonc
   {
     "compilerOptions": {
       "allowJs": true, "checkJs": true,
       "declaration": true, "emitDeclarationOnly": true,
       "declarationDir": "./types", "rootDir": "./src",
       "module": "nodenext", "moduleResolution": "nodenext",
       "target": "es2022", "lib": ["es2023"], "types": ["node"],
       "skipLibCheck": true,
       "strict": false, "strictNullChecks": true
     },
     "include": ["src/**/*.js"],
     "exclude": ["types", "node_modules"]   // exclude the output dir, or rebuilds nest under types/src
   }
   ```
3. `package.json`: `"typecheck": "tsc --noEmit"`, `"build:types": "tsc"`,
   `"prepublishOnly": "npm run build:types"`; `"types": "./types/index.d.ts"` +
   `"exports": { ".": { "types": "./types/index.d.ts", "default": "./src/index.js" } }`
   (`types` first); add `"types/"` to `files`.
4. `.gitignore`: add `/types/`.
5. `npm run typecheck`; fix what it surfaces — JSDoc gaps (comments only) and
   null-safety findings (minimal behavior-preserving guards; **never** `!`,
   `as any`, or `@ts-ignore`). Iterate to zero.
6. Shared/cross-file shapes: JSDoc `@typedef` in a `src/types.js` (generates its
   own `.d.ts`) — preferred over a hand-written `.d.ts`.
7. Untyped dep you **re-export types from**: minimal ambient `declare module`
   shim in a tracked file outside `./types`, shipped via `files`. Usually
   unnecessary under `strictNullChecks` (untyped imports collapse to `any`).
8. CI: push/PR workflow runs `typecheck` + `build:types` + tests; publish
   workflow runs `typecheck` before tests (`prepublishOnly` builds the types
   into the tarball at publish; keep publish idempotent).

## 3. The `<lib>.context.md` file (adopter-facing reference)

**What it does.** A single self-contained file that an adopter — or their AI
agent — reads to wire the library correctly *without* spelunking the source. It
ships via npm (listed in `files`) and is named `<lib>.context.md`.

**Why it's separate from the README.** The README is the pitch + the 6-line
quickstart — the hook. The `context.md` is the **complete contract**: every
option, the full public API, extension-point contracts, the scope boundaries
("what's NOT in X, and why"), gotchas, and the threat model. It's the file you
point an integrating agent at; it's where the durable design reasoning lives that
you don't want buried in a PRD the adopter never sees.

**Skeleton:** copy [`context.template.md`](context.template.md) to
`<lib>.context.md` and fill it. Sections (drop the ones that don't apply):

- `# <lib> — Integration Guide`
- `## What this is` — one paragraph, the problem it solves
- `## What <lib> is and is not` — the boundary, stated up front
- `## Which mode / components do I need?` — if there's more than one entry point
- `## Minimal usage` — the smallest working example (the "6-line" wiring)
- `## All options` — every config option, default, and what it does
- `## Public API` — exported functions/classes + their contracts
- `## <Extension> contract` — custom store/mailer/transport/etc. interfaces
- `## Architecture` — the shape, in a few sentences
- `## What's NOT in <lib>, and why` — the refusals, with the reasoning
- `## Threat model summary` — if security-relevant
- `## Gotchas` — the sharp edges adopters hit
- `## Constraints` — hard limits / non-negotiables

## 4. The doc set

| File | Ships? | Purpose |
|------|--------|---------|
| `README.md` | yes | Pitch + install + the 6-line quickstart. The hook. |
| `<lib>.context.md` | yes | Complete adopter contract (§3). Point agents here. |
| `GUIDE.md` / `OPS.md` | yes (if present) | Deeper how-to / operational runbook, as needed |
| `CHANGELOG.md` | yes | keep-a-changelog format; an entry every release |
| `CLAUDE.md` | **no** (repo-only) | Agent context: doctrine, most-litigated refusals, where rationale lives |
| `docs/PRD/SPEC/TASKS` | **no** (repo-only) | Product decisions, implementation contracts, task ledger |

Repo-only docs are excluded from `files` — they're for maintainers and agents,
not adopters. The adopter's complete picture is README + `context.md`.

## 5. CI shape

- **Push / PR** (`ci.yml`): `npm ci` → `typecheck` → `build:types` → `test`.
  Nothing else gates a merge. **No lint step** — `tsc` (`checkJs` +
  `strictNullChecks`) already catches the bug class that matters, and pulling in
  the ESLint ecosystem (devDeps + flat config + a new failure surface) cuts
  against the AGENT_RULES dependency hierarchy for a small vanilla-ESM lib.
  Formatting is a per-repo choice (e.g. Prettier as a dev-only convenience), not
  a merge gate.
- **Publish** (`publish.yml`): same gates before publish; `prepublishOnly`
  builds the types into the tarball; publish is idempotent (skip if the version
  is already on the registry) and verifies end-state rather than trusting npm's
  exit code. Authenticates with npm **trusted publishing (OIDC)** — no
  `NPM_TOKEN` — and is **manual** (`workflow_dispatch`): you publish when you
  want, not on every tag or merge.

The canonical workflow is [`publish.template.yml`](publish.template.yml) — copy
it to `.github/workflows/publish.yml`. (Configure the trusted publisher at
npmjs.com once before the first run.)
