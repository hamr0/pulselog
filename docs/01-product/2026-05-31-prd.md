## 1. Problem

The same apps (addypin, plato, gitdone) each grew **two** kinds of hand-written ops
code, re-implemented per app and drifting:

1. **"Is it up right now?"** ‚Äî a cron/timer script that curls `/health`, checks
   systemd units, disk, TLS expiry, mail queue; silent when green; emails on
   failure. Written ~3√ó (¬ß13).
2. **"How is it trending?"** ‚Äî an operator stats subsystem: a daily *collector*
   that snapshots foundational numbers to a JSONL log, and a weekly *digest* that
   reads the log and emails a week-over-week table. The **digest renderer is
   duplicated nearly verbatim** between `plato/bin/stats-weekly.js` and
   `gitdone/app/bin/stats-weekly.js`, and the **collector is rewritten per app**
   every time.

Both jobs are "observe a running app from outside, on a schedule, tell me only what
I need." flightlog records errors **from inside**; nothing generalizes the
**outside** view, and the per-app scripts can't share tooling because they don't
share a format. The operator keeps writing `stats.js` from scratch for each new app.

## 2. Goal

One zero-dependency, scheduled, **external** watcher ‚Äî the outside sibling to
flightlog ‚Äî that an app points a timer at. Two modes, one config, one JSONL dialect:

- **health** ‚Äî generalize the probe-and-alert script (¬ß6.2). Already built.
- **digest** ‚Äî generalize the stats subsystem so the operator **never writes a
  `stats.js` again**: declare the foundational numbers to watch, and pulselog owns
  the collect ‚Üí snapshot ‚Üí history ‚Üí WoW ‚Üí email pipeline (¬ß6.3). The *only* per-app
  customization is the metrics list; everything else is reusable.

Apps **port to pulselog after it is complete**; pulselog never edits their repos.

## 3. Success criteria

1. addypin's and plato's hand-written **health** scripts express fully as a pulselog
   `checks` config ‚Äî no bespoke bash left (¬ß13).
2. A new app gets weekly stats by declaring a **metrics list** in `digest` ‚Äî no
   `stats.js`, no `stats-weekly.js`. pulselog collects, accumulates, and emails the
   WoW table. Reproduces the existing plato/gitdone digest output.
3. The weekly email tells the operator not just *that* errors happened but **which
   kind** (api vs mail-auth) and flags any that crossed the threshold ‚Äî without
   emailing a single log line.
4. Zero production dependencies (`node:*` + global `fetch`), Node ‚â• 18.
5. Silent on green in health mode. Digest always sends (opt-in skip when flat).
6. **Only aggregate integers and error names ever leave the box; pulselog never
   emails message bodies or stacks** (¬ß4).
7. Ships JSDoc-generated types, a `pulselog.context.md` adopter contract, CI, and a
   manual OIDC publish ‚Äî same bar as flightlog (¬ß9).

## 4. What pulselog is NOT (the refusals *are* the product)

- **Not a daemon or scheduler.** You bring cron / a systemd timer. pulselog runs
  once and exits. History (for WoW) lives in the JSONL, not in a process.
- **Not a log aggregator / shipper / SIEM.** It does **not** collect, forward, tail
  into a store, or email raw logs. The digest reports *numbers and error names*,
  never log lines.
- **Not a metrics database / dashboard / TSDB.** It appends JSONL and renders a
  plain-text table. No UI, no server, no query layer, no charts. (The original sin.)
- **Not an uptime SaaS.** Local, no account, no telemetry, no phone-home.
- **Not an alerting platform.** One email per run. No routing, escalation, paging,
  on-call, dedup, or rate policy beyond "send once" / "skip if flat."
- **Not a transport.** Shipping the JSONL off-box is a **separate, consent-gated
  layer you build** (flightlog's `examples/ship.js` is the reference).
- **Not a PII collector.** pulselog stores **only the integer a metric command
  returns** ‚Äî never rows, never identifiers. "Unique customers" means *your* query
  does `count(distinct ‚Ä¶)`; pulselog records the integer and cannot leak what it
  never reads.
- **It never emails raw logs.** flightlog is **private by default** and stays on the
  box; the operator/VPS owner reads `errors.jsonl` there ‚Äî that is the point.
  pulselog's optional flightlog enrichment is an **aggregate**: the 7-day error
  **count**, the **top error names with counts** (so you can tell *which* area is
  noisy ‚Äî api vs mail-auth ‚Äî not just *that* something failed), and a per-name **‚â•N
  flag**. Names + counts are aggregates the operator already owns; **message bodies
  and stacks are never emailed** (they carry payloads/PII) ‚Äî read those on the box.
- **Not flightlog.** flightlog captures from *inside*; pulselog observes from
  *outside*. Separate tools, separate files, one shared dialect.
- **No HTTP body assertion.** An `http` check verifies the **status code only**.
  plato's `db_writable` goes through the `command` escape hatch ‚Äî pulselog core never
  grows JSON/body parsing (¬ß12, the original sin).

## 5. Decisions (locked)

| # | Decision | Why |
|---|----------|-----|
| D1 | **One tool, two modes** ‚Äî `health` (default) and `digest` (`--digest`). | Same engine: a check is `command ‚Üí pass/fail`, a metric is `command ‚Üí integer`. Both reuse `sink.js` + `email.js`. |
| D2 | **Modes stay crisp.** Separate config sections (`checks` vs `digest`), separate invocation. | The chatty weekly digest must never erode health's *silent-on-green* discipline. |
| D3 | **pulselog owns collection too.** The adopter declares a **metrics list** (foundational numbers + the command that yields each); pulselog does collect ‚Üí snapshot ‚Üí history ‚Üí WoW ‚Üí email. | The whole point: stop rewriting `stats.js`/`stats-weekly.js` per app. Lean customization, reusable everything-else. |
| D4 | **One weekly run = collect + digest.** Snapshots once (**one line per week**), appends to `history`, then renders the last `weeks` weeks. | Leanest: one timer, no DB/lockfile. "One line per week + history." |
| D5 | **Metric source = a `command` that prints one integer.** | Same escape hatch as the health `command` check; SQLite `select count(...)` covers the apps. App-specific logic lives in *your* query, not pulselog. Integers only. |
| D6 | **Digest always sends; opt-in `skipIfFlat`.** | A report, not an alert. Skip only when every Œî is 0 **and** no flightlog flag. |
| D7 | **flightlog enrichment optional; aggregate count + top error names + ‚â•N flag, weekly (7d).** Grouped by a configurable field (default the error `name`). **Persisted in the weekly history line and shown in the email ‚Äî never messages/stacks.** | Privacy (¬ß4): tells you *which* area without emailing logs. flightlog stays private/local. |
| D8 | **Alert transport = system `mail`/`sendmail`; suggest `msmtp` as an addon.** | Zero-dep; the user chooses the transport their box provides. |
| D9 | **One config, one source of truth.** `checks` + `digest` in a single `pulselog.config.json`. | Clean/lean; no scattered files. |
| D10 | **Exit 0 on a health/stat failure; exit 1 only on a run/config error.** | Cron quiet on a normal failure; a misconfig surfaces loudly. |

## 6. Surface (the whole contract)

### 6.1 CLI

```
pulselog --config ./pulselog.config.json            # health (default): run checks, silent on green
pulselog --digest --config ./pulselog.config.json   # digest: snapshot metrics ‚Üí WoW table ‚Üí email
pulselog --digest --dry-run --config ‚Ä¶              # render to stdout, don't send/append
```

### 6.2 Health mode (built)

`checks: [{ type, name, enabled, ‚Ä¶fields }]`, each `async (cfg) ‚Üí {ok, reason, detail?}`,
never throwing: `http` (status code only), `tcp`, `ssl`, `disk`, `file-age`,
`service`, `command`. Runs enabled checks concurrently ‚Üí one `{kind:"health",
status:"fail", ‚Ä¶}` JSONL line per failure ‚Üí **one** summary email if anything
failed. Silent on green. Validated in ¬ß13.

### 6.3 Digest mode (designed)

```jsonc
"digest": {
  "app": "addypin",
  "history": "/var/lib/addypin/stats.jsonl",   // pulselog writes + reads this (one line/week, the record)
  "email": "ops@addypin.com",                  // omit / --dry-run ‚Üí print, no send
  "from": "alerts@addypin.com",
  "weeks": 4,                                    // rows in the table (default 4)
  "skipIfFlat": false,                           // true ‚Üí no email when every Œî=0 and no flag
  "metrics": [                                   // ‚Üê the ONLY per-app customization
    { "name": "users", "command": "sqlite3", "args": ["‚Ä¶/addypin.db", "select count(distinct customer) from pins"] },
    { "name": "pins",  "command": "sqlite3", "args": ["‚Ä¶/addypin.db", "select count(*) from pins"] }
  ],
  "flightlog": { "file": "/var/lib/addypin/errors.jsonl", "groupBy": "name", "flagAtLeast": 20 }  // optional
}
```

One weekly run:
1. Run each metric `command` ‚Üí trim stdout, parse **one integer** (non-integer /
   failed ‚Üí record `null` + note; never sinks the run).
2. If `flightlog.file` set: read it, count errors in the **past 7 days**, group by
   `flightlog.groupBy` (default the error `name`), keep the top few, flag any group
   whose 7-day count ‚â• `flagAtLeast` (default 20). **Names + counts only.**
3. Append **one snapshot line** (mode `0600`) to `history` ‚Äî the canonical weekly
   record (metrics **and** the error summary), so trends are kept for reference:
   ```json
   {"ts":"2026-05-31T06:00:00Z","kind":"stats","app":"addypin","users":2,"pins":5,
    "errors":{"total_7d":31,"top":{"ApiTimeout":24,"SmtpAuthError":7},"flagged":["ApiTimeout"]}}
   ```
4. Read `history`, **ISO-week bucket** (Mon‚ÄìSun, the proven algorithm), keep the
   latest snapshot per week, take the last `weeks` ‚Üí render the fixed-width **WoW
   table** (Œî at render against the next-older row; blank when 0 / no prior).
5. If `email` set and not `skipIfFlat`-skipped: email the table + the flightlog line.
   Else the snapshot line is the artifact.

Email (reproduces the plato/gitdone digest + the actionable error line):
```
addypin weekly stats ‚Äî 2026-W18 ‚Üí 2026-W22
snapshots in log: 22

week     | users     Œî | pins      Œî
---------------------------------------
2026-W22 |     2       |     5
2026-W21 |     2       |     5
‚Ä¶
flightlog (last 7d): 31 errors. top: ApiTimeout√ó24, SmtpAuthError√ó7.   ‚â•20: ApiTimeout
```

Generic by construction: plato declares `users, posts`; gitdone `events`. pulselog
never knows what they mean. The error line answers "api or mail-auth?" because it
groups by the error name (or a context field you set via `groupBy`) ‚Äî not a bare count.

## 7. The shared JSONL dialect

flightlog core fields (`ts`, `kind`, ‚Ä¶) so one toolset reads all streams; each
writer owns its **own** file. `kind`: `uncaught`/`manual` (flightlog), `health`
(pulselog health), `stats` (pulselog digest). Compose at read time:
`jq -s 'sort_by(.ts)' errors.jsonl health.jsonl stats.jsonl`.

## 8. Self-failure handling

A check/metric that errors is caught and recorded (failure / `null`) ‚Äî one broken
probe never sinks the run. The sink swallows write failures and warns once to stderr.
A digest run with no readable `history` starts fresh (first week = baseline, no Œî);
a newly-added metric shows baseline until it has a prior week.

## 12. Open / deferred

- **`http` JSON-body assertion** ‚Äî **dropped, not deferred.** http is status-only.
- **flightlog `groupBy` semantics** ‚Äî default is the error `name` (class). If an
  app's "api vs mail-auth" distinction lives in a flightlog `context` field instead
  (e.g. `where`), set `groupBy` to that key. **CONFIRM** which field carries the area
  in the apps' `errors.jsonl` (see Q at end). Configurable, lean ‚Äî no parsing logic.
- **HTTP/JSON metric source** ‚Äî a metric is a `command ‚Üí integer` (covers all current
  SQLite apps). Add an HTTP source only if a number lives behind an endpoint. Deferred.
- **Daily collector + separate weekly digest (two timers)** ‚Äî pulselog 0.1.0 does
  one weekly run for leanness. Split only if an adopter needs intra-week granularity.
- **Richer trend math** (slopes, anomalies) ‚Äî out; a dashboard's job (¬ß4).
