**North star (the doctrine that governs every call):** pulselog is a **lightweight
wrapper of what the server/OS already offers** — reusable, consistent, simple. We
never stoop into complicating features, adding surface, or a GUI. Complexity is the
original sin. Default new API surface OUT; the burden of proof is on adding.

---

## 1. Problem

The same apps (addypin, plato, gitdone) each grew **two** kinds of hand-written ops
code, re-implemented per app and drifting:

1. **"Is it up right now?"** — a cron/timer script that curls `/health`, checks
   systemd units, disk, TLS expiry, mail queue; silent when green; emails on
   failure. Written ~3× (§13).
2. **"How is it trending?"** — an operator stats subsystem: a daily *collector*
   that snapshots foundational numbers to a JSONL log, and a weekly *digest* that
   reads the log and emails a week-over-week table. The **digest renderer is
   duplicated nearly verbatim** between `plato/bin/stats-weekly.js` and
   `gitdone/app/bin/stats-weekly.js`, and the **collector is rewritten per app**
   every time.

Both jobs are "observe a running app from outside, on a schedule, tell me only what
I need." flightlog records errors **from inside**; nothing generalizes the
**outside** view, and the per-app scripts can't share tooling because they don't
share a format. The operator keeps writing `stats.js` from scratch for each new app.

## 2. Goal

One zero-dependency, scheduled, **external** watcher — the outside sibling to
flightlog — that an app points a timer at. Three modes, one config, one JSONL dialect:

- **health** — generalize the probe-and-alert script (§6.2). Already built.
- **digest** — generalize the stats subsystem so the operator **never writes a
  `stats.js` again**: declare the foundational numbers to watch, and pulselog owns
  the collect → snapshot → history → WoW → email pipeline (§6.3). The *only* per-app
  customization is the metrics list; everything else is reusable.
- **backup** — generalize the per-app `*-backup.sh` so the operator **never re-writes
  the tar-and-rotate envelope again**: declare a dump command + a rolling window, and
  pulselog owns stage → `tar` → atomic write → retention → integrity floor → JSONL +
  alert (§6.4). The *only* per-app customization is the dump command; everything else
  is reusable. (Designed; targets `0.2.0`.)

Apps **port to pulselog after it is complete**; pulselog never edits their repos.

## 3. Success criteria

1. addypin's and plato's hand-written **health** scripts express fully as a pulselog
   `checks` config — no bespoke bash left (§13).
2. A new app gets weekly stats by declaring a **metrics list** in `digest` — no
   `stats.js`, no `stats-weekly.js`. pulselog collects, accumulates, and emails the
   WoW table. Reproduces the existing plato/gitdone digest output.
3. The weekly email tells the operator not just *that* errors happened but **which
   kind** (api vs mail-auth) and flags any that crossed the threshold — without
   emailing a single log line.
4. Zero production dependencies (`node:*` + global `fetch`), Node ≥ 18.
5. Silent on green in health mode. Digest always sends (opt-in skip when flat).
6. **Only aggregate integers and error names ever leave the box; pulselog never
   emails message bodies or stacks** (§4).
7. Ships JSDoc-generated types, a `pulselog.context.md` adopter contract, CI, and a
   manual OIDC publish — same bar as flightlog (§9).

## 4. What pulselog is NOT (the refusals *are* the product)

- **Not a daemon or scheduler.** You bring cron / a systemd timer. pulselog runs
  once and exits. History (for WoW) lives in the JSONL, not in a process.
- **Not a log aggregator / shipper / SIEM.** It does **not** collect, forward, tail
  into a store, or email raw logs. The digest reports *numbers and error names*,
  never log lines.
- **Not a metrics database / dashboard / TSDB.** It appends JSONL and renders a
  plain-text table. No UI, no server, no query layer, no charts. (The original sin.)
- **Not an uptime SaaS.** Local, no account, no telemetry, no phone-home.
- **Not an alerting platform.** One email per run. No routing, escalation, paging,
  on-call, dedup, or rate policy beyond "send once" / "skip if flat."
- **Not a transport.** Shipping the JSONL off-box is a **separate, consent-gated
  layer you build** (flightlog's `examples/ship.js` is the reference).
- **Not a PII collector.** pulselog stores **only the integer a metric command
  returns** — never rows, never identifiers. "Unique customers" means *your* query
  does `count(distinct …)`; pulselog records the integer and cannot leak what it
  never reads.
- **It never emails raw logs.** flightlog is **private by default** and stays on the
  box; the operator/VPS owner reads `errors.jsonl` there — that is the point.
  pulselog's optional flightlog enrichment is an **aggregate**: the 7-day error
  **count**, the **top error names with counts** (so you can tell *which* area is
  noisy — api vs mail-auth — not just *that* something failed), and a per-name **≥N
  flag**. Names + counts are aggregates the operator already owns; **message bodies
  and stacks are never emailed** (they carry payloads/PII) — read those on the box.
- **Not flightlog.** flightlog captures from *inside*; pulselog observes from
  *outside*. Separate tools, separate files, one shared dialect.
- **No HTTP body assertion.** An `http` check verifies the **status code only**.
  plato's `db_writable` goes through the `command` escape hatch — pulselog core never
  grows JSON/body parsing (§12, the original sin).
- **Not a backup *engine*.** Backup mode wraps the **adopter's** dump command; pulselog
  knows nothing about databases — no `pg_dump`/`VACUUM`/engine drivers ever enter core
  (the original sin). It guarantees only *"the dump exited 0 and produced an archive ≥
  `minBytes`"* — **not** that the archive restores. Restore-testing, off-host copy
  (your `rsync` / SSH keys), and encryption stay the operator's; pulselog owns the
  envelope, not the data semantics (§6.4).

## 5. Decisions (locked)

| # | Decision | Why |
|---|----------|-----|
| D1 | **One tool, three modes** — `health` (default), `digest` (`--digest`), `backup` (`--backup`). | One engine: a check is `command → pass/fail`, a metric is `command → integer`, a backup is `command → files`. All reuse `sink.js` + `email.js`. (backup → `0.2.0`.) |
| D2 | **Modes stay crisp.** Separate config sections (`checks` vs `digest`), separate invocation. | The chatty weekly digest must never erode health's *silent-on-green* discipline. |
| D3 | **pulselog owns collection too.** The adopter declares a **metrics list** (foundational numbers + the command that yields each); pulselog does collect → snapshot → history → WoW → email. | The whole point: stop rewriting `stats.js`/`stats-weekly.js` per app. Lean customization, reusable everything-else. |
| D4 | **One weekly run = collect + digest.** Snapshots once (**one line per week**), appends to `history`, then renders the last `weeks` weeks. | Leanest: one timer, no DB/lockfile. "One line per week + history." |
| D5 | **Metric source = a `command` that prints one integer.** | Same escape hatch as the health `command` check; SQLite `select count(...)` covers the apps. App-specific logic lives in *your* query, not pulselog. Integers only. |
| D6 | **Digest always sends; opt-in `skipIfFlat`.** | A report, not an alert. Skip only when every Δ is 0 **and** no flightlog flag. |
| D7 | **flightlog enrichment optional; aggregate count + top error names + ≥N flag, weekly (7d).** Grouped by a configurable field (default the error `name`). **Persisted in the weekly history line and shown in the email — never messages/stacks.** | Privacy (§4): tells you *which* area without emailing logs. flightlog stays private/local. |
| D8 | **Alert transport = system `mail`/`sendmail`; suggest `msmtp` as an addon.** | Zero-dep; the user chooses the transport their box provides. |
| D9 | **One config, one source of truth.** `checks` + `digest` in a single `pulselog.config.json`. | Clean/lean; no scattered files. |
| D10 | **Exit 0 on a health/stat failure; exit 1 only on a run/config error.** | Cron quiet on a normal failure; a misconfig surfaces loudly. |
| D11 | **Backups are a third *mode*, not a sibling lib.** A backup-only package is too thin to stand and too much maintenance, with nothing to pair it with. As a mode it reuses `sink.js`/`email.js`/config/CLI/dialect — adopters import nothing new. | Earns its place: the tar-and-rotate **envelope** is hand-rolled ≥3× (addypin/plato/gitdone `*-backup.sh`) — the same ≥3× duplication that justified absorbing the digest (D3, §13). |
| D12 | **Envelope owns the lifecycle; sources are curated-or-custom** *(refined by D16/D17)*. pulselog owns stage → `tar` → atomic write → rolling retention → integrity floor → `kind:"backup"` line → alert, and stages three source kinds into `$PULSELOG_STAGE`: built-in safe-default DB dumps (`db`, D16), a static file copy (`include`, D17), and the `command` opt-out. | Same mechanism/policy split as a metric or a health `command`. pulselog grows curated *safe defaults* for the common OSS engines (the consistency opinion is the value), but never a general engine matrix — exotic stores stay the adopter's `command`. |
| D13 | **Rolling retention by `keepLast` (count) and/or `keepDays` (age).** Deletes **only** files matching pulselog's own `<name>-<stamp>.tar.gz` in the configured `dir`, and **never rotates when the new archive failed/empty.** | plato keeps newest-N, gitdone keeps-by-age — both are in the fleet. Bounding the one destructive action (own prefix, own dir, never on a failed run) keeps a bad backup from deleting good ones. |
| D14 | **Integrity floor + honesty.** Atomic `.tmp`→`mv` (a half-written tar never appears under the canonical name); the archive must exist and be ≥ `minBytes` or the run fails. pulselog asserts the dump ran and produced a non-trivial archive — **not** restorability. | "`command` exited 0" alone is false confidence (a truncated `pg_dump` exits 0 under disk-full). The size floor + recorded `bytes` make a broken backup *visible* without pulselog pretending to verify a restore. |
| D15 | **A failed backup exits `1` (loud) — unlike a failed health check.** *(RATIFIED 2026-05-31)* | Health's job is to *observe*; finding a service down is success (exit 0, D10). Backup's job is to *produce* the artifact; failing to produce it is the run failing, like a bad config — and a silently-missing backup is the one failure that must never be quiet. (A health `file-age` check on `dir` is the belt-and-suspenders net.) |
| D16 | **Built-in safe-default DB dumps for the common OSS engines** (0.2.0: `sqlite` → node:sqlite `VACUUM INTO`; `postgres` → `pg_dump -Fc`; `mysql`/MariaDB → `mysqldump --single-transaction`), with `command` as the opt-out. *(RATIFIED 2026-05-31)* | "It's just your `command`" is under-opinionated for the one mode where being wrong is invisible-until-restore. Even two SQLite apps in the fleet dump differently (plato `VACUUM INTO` via node:sqlite to dodge the too-old system CLI; addypin copies `.db`+`-shm`+`-wal`). The default *encodes the consistency opinion* (`--single-transaction`, `-Fc`, online VACUUM). Not an engine matrix: pg/mysql shell to the standard tools, sqlite uses a Node builtin, missing/wrong-version fails loud, exotic stores stay `command`. |
| D17 | **`include`: a generic static-file copy into the stage** (certs/configs/keys), each entry required by default (missing → fail loud, D15) or `{path,optional:true}` (skip + record in the line); **symlinks preserved**. | The static `tar /etc/letsencrypt` / `cp config` half is hand-rolled in all three `*-backup.sh` (addypin certs; gitdone certs+DKIM keys+env; plato posts+config+spam) — generic, no app knowledge, earns absorption. Required-by-default keeps a silently-incomplete backup from passing; `optional` covers fresh-install gaps (plato skips an absent `config.json`). Preserving symlinks keeps `/etc/letsencrypt` `live/→archive/` restorable. |
| D18 | **The backup payload is treated as secret-bearing.** Archive `chmod 0600`, `dir`/stage `0700`; DB passwords go via env (`PGPASSWORD`/`MYSQL_PWD`), never argv (stripped from URLs); same-name source collisions fail loud; email header fields are single-line (no injection). *(from the security audit; fixed pre-`0.2.0`)* | The archive is the most sensitive artifact pulselog makes — DB dumps + TLS/DKIM keys — yet the original cut left it `0644` and passed a postgres password in `pg_dump` argv (`ps`-readable). On a shared host that's local disclosure of everything backed up. The fixes match the sink's existing `0600` posture and the mysql path's env-based password handling; each was validated with a repro, fixed, and verified against real Postgres + MariaDB. |

## 6. Surface (the whole contract)

### 6.1 CLI

```
pulselog --config ./pulselog.config.json            # health (default): run checks, silent on green
pulselog --digest --config ./pulselog.config.json   # digest: snapshot metrics → WoW table → email
pulselog --digest --dry-run --config …              # render to stdout, don't send/append
```

### 6.2 Health mode (built)

`checks: [{ type, name, enabled, …fields }]`, each `async (cfg) → {ok, reason, detail?}`,
never throwing: `http` (status code only), `tcp`, `ssl`, `disk`, `file-age`,
`service`, `command`. Runs enabled checks concurrently → one `{kind:"health",
status:"fail", …}` JSONL line per failure → **one** summary email if anything
failed. Silent on green. Validated in §13.

### 6.3 Digest mode (designed)

```jsonc
"digest": {
  "app": "addypin",
  "history": "/var/lib/addypin/stats.jsonl",   // pulselog writes + reads this (one line/week, the record)
  "email": "ops@addypin.com",                  // omit / --dry-run → print, no send
  "from": "alerts@addypin.com",
  "weeks": 4,                                    // rows in the table (default 4)
  "skipIfFlat": false,                           // true → no email when every Δ=0 and no flag
  "metrics": [                                   // ← the ONLY per-app customization
    { "name": "users", "command": "sqlite3", "args": ["…/addypin.db", "select count(distinct customer) from pins"] },
    { "name": "pins",  "command": "sqlite3", "args": ["…/addypin.db", "select count(*) from pins"] }
  ],
  "flightlog": { "file": "/var/lib/addypin/errors.jsonl", "groupBy": "name", "flagAtLeast": 20 }  // optional
}
```

One weekly run:
1. Run each metric `command` → trim stdout, parse **one integer** (non-integer /
   failed → record `null` + note; never sinks the run). Optionally declare
   `metricsCommand` — one command emitting a **flat JSON object of named integers** in
   a single pass; a metric *without* its own `command` takes its value by `name` from
   that object (a metric *with* a `command` still runs and overrides it). Amortizes an
   expensive snapshot; the contract is unchanged — only declared names reach the line,
   each value passes the same whole-number gate. *(added post-`0.2.0`, first-adopter
   feedback: gitdone computes ~14 metrics in one scan.)*
2. If `flightlog.file` set: read it, count errors in the **past 7 days**, group by
   `flightlog.groupBy` (default the error `name`), keep the top few, flag any group
   whose 7-day count ≥ `flagAtLeast` (default 20). **Names + counts only.**
3. Append **one snapshot line** (mode `0600`) to `history` — the canonical weekly
   record (metrics **and** the error summary), so trends are kept for reference:
   ```json
   {"ts":"2026-05-31T06:00:00Z","kind":"stats","app":"addypin","users":2,"pins":5,
    "errors":{"total_7d":31,"top":{"ApiTimeout":24,"SmtpAuthError":7},"flagged":["ApiTimeout"]}}
   ```
4. Read `history`, **ISO-week bucket** (Mon–Sun, the proven algorithm), keep the
   latest snapshot per week, take the last `weeks` → render the fixed-width **WoW
   table** (Δ at render against the next-older row; blank when 0 / no prior).
5. If `email` set and not `skipIfFlat`-skipped: email the table + the flightlog line.
   Else the snapshot line is the artifact.

Email (reproduces the plato/gitdone digest + the actionable error line):
```
addypin weekly stats — 2026-W18 → 2026-W22
snapshots in log: 22

week     | users     Δ | pins      Δ
---------------------------------------
2026-W22 |     2       |     5
2026-W21 |     2       |     5
…
flightlog (last 7d): 31 errors. top: ApiTimeout×24, SmtpAuthError×7.   ≥20: ApiTimeout
```

Generic by construction: plato declares `users, posts`; gitdone `events`. pulselog
never knows what they mean. The error line answers "api or mail-auth?" because it
groups by the error name (or a context field you set via `groupBy`) — not a bare count.

### 6.4 Backup mode (designed → `0.2.0`) — RATIFIED 2026-05-31

```jsonc
"backup": {
  "app": "plato",
  "dir": "/var/lib/plato/backups",          // archives live here (its OWN dir); pulselog stages, tars, and rotates within it
  "name": "plato-backup",                    // archive prefix → <name>-<UTC stamp>.tar.gz

  // (A) db — built-in SAFE-DEFAULT dumps for the common OSS engines (sqlite, postgres).
  //     pulselog runs the consistency-correct dump INTO $PULSELOG_STAGE. Object or array.
  "db": [
    { "engine": "sqlite",   "path": "/var/lib/plato/forum.db" },                 // → node:sqlite VACUUM INTO (zero-dep, online, checkpoints WAL)
    { "engine": "sqlite",   "path": "/var/lib/plato/knowless.db", "optional": true },
    { "engine": "postgres", "url": "postgres://u@/plato", "passwordEnv": "PGPASSWORD" }, // → pg_dump -Fc
    { "engine": "mysql",    "url": "mysql://u@/app",    "passwordEnv": "MYSQL_PWD" }     // → mysqldump --single-transaction
  ],

  // (B) include — STATIC sources pulselog copies into the stage (certs/configs/keys).
  //     String = required (missing → fail loud, D15); {path,optional:true} = skip+record. Symlinks preserved.
  "include": [
    "/var/lib/plato/posts",
    "/var/lib/plato/config.json",
    { "path": "/etc/opendkim/keys", "optional": true }
  ],

  // (C) command — the OPT-OUT escape hatch: bring your own dump (mongodump, redis BGSAVE,
  //     a replica, anything) writing into $PULSELOG_STAGE. Runs after db + include staging.
  "command": "node", "args": ["bin/custom-dump.mjs"],

  "keepLast": 7,                             // rolling window by count (plato) …
  "keepDays": 30,                            // … and/or by age (gitdone). At least one required.
  "minBytes": 1024,                          // integrity floor — a smaller archive fails the run
  "history": "/var/lib/plato/backup.jsonl",  // one kind:"backup" line per run (its OWN file)
  "email": "ops@plato…", "from": "alerts@…"  // alert on failure; omit → the JSONL line is the record
}
```
**At least one source** (`db` | `include` | `command`) is required. plato/addypin go
fully declarative with `db`+`include` and **no `command` at all** (see D16).

**Built-in `db` engines (0.2.0) — the foundational OSS relational trio.** Each safe
default encodes the *consistency* opinion; the tool must be on `PATH` (else the run
fails loud) except sqlite, which is in-process:

| `engine` | Safe default | Output in stage | Connection |
|---|---|---|---|
| `sqlite` | `node:sqlite` `VACUUM INTO` (bundled, online, checkpoints WAL) | `<label>.db` | `path` |
| `postgres` | `pg_dump -Fc` (custom, compressed, selective restore) | `<label>.dump` | `url` (+ `passwordEnv`) |
| `mysql` | `mysqldump --single-transaction --quick --routines --triggers` (MySQL **and** MariaDB) | `<label>.sql` | `url` (+ `passwordEnv`) |

`label` defaults to the engine + an index (`sqlite-1`) or is set per entry via `name`.
MongoDB (SSPL — not OSS), Redis (RSAL/SSPL; OSS fork = Valkey), and anything else stay
the `command` opt-out + the dump cookbook in `pulselog.context.md`.

One scheduled run (a nightly timer, separate from health/digest):
1. Make a fresh staging dir, `$PULSELOG_STAGE`. Stage the sources **into** it, in order:
   - **`db`** — for each entry run pulselog's safe-default dump (sqlite → node:sqlite
     `VACUUM INTO`; postgres → `pg_dump -Fc`; mysql → `mysqldump --single-transaction
     --result-file`, streamed to disk — never buffered in memory) into
     `$PULSELOG_STAGE/<label>`. The DB **password is passed via env** (`PGPASSWORD`/
     `MYSQL_PWD`), stripped from any URL so it never lands in argv (process-table leak,
     D18). A `db` entry that omits its connection field (`path` for sqlite, `url` for
     postgres/mysql) or names an engine outside the built-in trio → **fail loud with the
     field/engine named** (no half-empty archive). A required dump that errors → the run
     fails (exit 1); an `optional` db whose file is absent → skip+record. Two sources that
     would stage under the same name → fail loud.
   - **`include`** — copy each path into the stage, **symlinks preserved** (so
     `/etc/letsencrypt` `live/→archive/` restores intact). A required path that's
     missing → fail loud (exit 1); an `optional` one → skip and record in the line.
   - **`command`** — if set, run `command`/`args` with `$PULSELOG_STAGE` set (and a
     timeout). Non-zero exit → the run fails (exit 1).
2. `tar -czf <dir>/<name>-<stamp>.tar.gz.tmp -C $PULSELOG_STAGE .`, **`chmod 0600`**
   the archive (it holds DB dumps + private keys — owner-only, never group/world), then
   **atomic `mv`** to the canonical name — a failed/partial tar never appears under it.
   (`dir` is created `0700`; the staging dir is `0700`.)
3. **Integrity floor:** the archive exists and is ≥ `minBytes`, else fail (alert, exit
   1) **without rotating** — a bad run never deletes a good prior backup.
4. **Rolling retention:** among `<dir>/<name>-*.tar.gz` only, keep the newest
   `keepLast` and/or those newer than `keepDays`; delete the rest.
5. Append **one** `{kind:"backup"}` line (mode `0600`) to `history` — its own file:
   ```json
   {"ts":"2026-05-31T02:00:05Z","kind":"backup","app":"plato","name":"plato-backup","status":"ok","bytes":48213050,"files":5,"durationMs":3120,"kept":7,"skipped":[]}
   ```
6. Success → silent (the JSONL line is the record). Failure → one email, exit 1 (D15).

This is exactly the *tail* of every `*-backup.sh` (mktemp staging → tar → atomic mv →
keep-N / keep-days rotation → alert), lifted out once and tested. The app-specific
*head* is split: the **common-engine dumps are curated safe defaults pulselog owns**
(`db`, D16), static files are a generic copy (`include`), and anything exotic stays the
adopter's `command` opt-out. pulselog owns the archive lifecycle; it stores only
sizes/counts, never what's inside (privacy, §4).

**The triad:** `health` (is it up) · `stats` (how's it trending) · `backup` (is it
safe). A backup's `bytes` can itself become a digest metric (size trend), and a health
`file-age` check on `dir` catches "backups silently stopped" as the safety net — three
modes, one config, one dialect, composing at read time.

## 7. The shared JSONL dialect

flightlog core fields (`ts`, `kind`, …) so one toolset reads all streams; each
writer owns its **own** file — **never one shared file** (two processes appending +
rotation is the race D2/§8 forbid, and the modes run on different timers anyway). One
*dialect*, not one file. `kind`: `uncaught`/`manual` (flightlog), `health` (pulselog
health), `stats` (pulselog digest), `backup` (pulselog backup). Compose at read time:
`jq -s 'sort_by(.ts)' errors.jsonl health.jsonl stats.jsonl backup.jsonl`.

## 8. Self-failure handling

A check/metric that errors is caught and recorded (failure / `null`) — one broken
probe never sinks the run. The sink swallows write failures and warns once to stderr.
A digest run with no readable `history` starts fresh (first week = baseline, no Δ);
a newly-added metric shows baseline until it has a prior week.

- **`http` JSON-body assertion** — **dropped, not deferred.** http is status-only.
- **flightlog `groupBy` semantics** — default is the error `name` (class). If an
  app's "api vs mail-auth" distinction lives in a flightlog `context` field instead

  in the apps' `errors.jsonl` (see Q at end). Configurable, lean — no parsing logic.
- **HTTP/JSON metric source** — a metric is a `command → integer` (covers all current
  SQLite apps). Add an HTTP source only if a number lives behind an endpoint. Deferred.
- **Daily collector + separate weekly digest (two timers)** — pulselog 0.1.0 does
  one weekly run for leanness. Split only if an adopter needs intra-week granularity.
- **Richer trend math** (slopes, anomalies) — out; a dashboard's job (§4).
- **Backup mode targets `0.2.0`** — *RATIFIED* (moot now: `0.1.0` shipped, so backup
  lands in `0.2.0` as planned).
- **Backup failure exit code = `1` (loud), per D15** — *RATIFIED 2026-05-31*.
- **Backup `$PULSELOG_STAGE` handoff** (staging dir, tar what lands) — *RATIFIED 2026-05-31*
  (vs. tarring a fixed path).
- **Backup `db` safe-default engines = sqlite + postgres + mysql/MariaDB** (D16); `include`
  static-copy with per-path `optional` (D17) — *RATIFIED 2026-05-31*.
- **Backup off-host copy + encryption + restore-test** — explicitly **out** of pulselog
  (the operator's `rsync` / keys / policy). pulselog owns the local envelope only (§4, D14).
  *Recommended shape (`0.3.1` docs, not a feature):* the off-host step is a **pull, not a
  push** — the backed-up box holds no credential to the off-host copy, so a compromised
  box can't delete the backup it can't write — via a **read-only forced-command SSH key**
  (`examples/pull-restricted.sh`), watched off-host with a `file-age` dead-man's-switch.
  A post-publish push hook was considered and **declined** (push is weaker for the
  compromise case; the `&&`-chain + pull already cover it — revisit only if many ask).
- **Backup engines beyond the trio (Mongo/Redis/…)** — `command` opt-out + a dump cookbook
  in `pulselog.context.md`. Deferred as built-ins (Mongo SSPL / Redis RSAL aren't OSS; Valkey TBD).
- **Per-check `timeoutMs` + in-run retry** (`0.4.0`, first-adopter feedback) — `service`/`disk`
  un-hardcode their 5 s timeout (defaults unchanged) and label a killed probe as `timeout`;
  optional `retries`/`retryDelayMs` (per-check or a global `retry` default) re-probe a failing
  check within one run so a transient blip on a shared host doesn't page. **Mechanism** (probe
  accuracy + timeout) is pulselog's; **policy** (how many failures = a page) stays the adopter's.
  *Cross-run consecutive-failure debounce declined* — it needs persistent health state and is
  alert dedup (a stated refusal); threshold-paging belongs to the JSONL-consuming layer.
- **Security pass** (`0.4.0`, audit + repro-validated) — the CLI refuses a group/world-writable or
  non-owner config (it executes commands as us, often root); backup tightens its umask + enforces
  `dir` 0700 even when pre-existing (no temp-archive perms window); the DB password reaches only the
  engine's own env var; and a `name`/`db[].name` with `/`/`\`/`..` is rejected (can't escape the
  stage). Config remains the documented trust boundary; these are defense-in-depth, each repro'd.
- **Config-gate refinement** (`0.4.1`, first-adopter feedback) — the `0.4.0` ownership rule was
  stricter than the `ssh`/`sudo` precedent it cited: it refused a root-owned config read by a
  non-root service user, even though that file isn't writable by the reader and so isn't
  attacker-controlled. The gate now allows an owner of **the running user or root** (still refusing
  group/world-writable), matching `ssh` exactly and letting a root-owned deploy tree serve service
  units — the common shape the first adopter (gitdone) hit on upgrade. One line; no default change.

## 15. Fallback alert sink (`0.7.0`) — RATIFIED & SHIPPED 2026-07-02

**Origin.** The 2026-07 addypin mail-delivery incident: `mail.<domain>` was CDN-proxied,
breaking SPF/rDNS, so **every** outbound message (health alerts *and* the digest) bounced
at Gmail for ~a month. pulselog ran, detected, tried to email — operator got nothing. The
**circular-alert gap**: the one alert that says "mail is broken" rides the same broken path.
(Absorbed from the addypin proposal `docs/03-logs/pulselog-fallback-alert-proposal.md`.)

**Shape.** Opt-in `fallback` block on each alert-bearing config (`alert`, `digest`,
`backup`) — a second, out-of-band delivery path. Mechanism in the lib, policy (which sink)
in the config. Purely additive: absent `fallback` → byte-for-byte prior behavior.

**Decisions (locked).**
- **D1 — command sink only; no native webhook.** Both are zero-dep, but the command sink
  reuses the spawn pulselog already has; a webhook is a second way to do what `curl` does
  (AGENT_RULES: reuse primitives, don't write new code where similar exists). Deferred.
- **D2 — default `when: "always"`.** The incident's handoff succeeded (exit 0) while the
  bounce was async — so `on-primary-failure` would have stayed silent. `always` is the only
  mode that closes the gap, hence the default; `on-primary-failure` remains for handoff-only.
- **D3 — covers all three sites** (health, digest, backup). A silently-missing backup is
  the worst case (D15) → it needs the fallback most.
- **D4 — best-effort, never fatal.** A broken sink is recorded, never throws, never changes
  the exit code (validated by an A/B control: the same input crashes an unguarded process,
  exit 1, and is neutralized by the wrapper, exit 0).
- **D5 — the JSONL records each sink's outcome** (`kind:"alert"` line for health; a
  `fallback` field on the backup fail record; `delivered:"fallback"` from the digest).
- **D6 — privacy holds** — the fallback carries the already-redacted render (counts + names
  only for the digest), never the raw data.
- **D7 — `--dry-run` skips both sinks.**
- **D8 — one shared `spawnDelivery` primitive** (`src/deliver.js`) serves the `mail`/
  `sendmail` sends *and* the fallback: "spawn a delivery command, pipe the body to stdin,
  best-effort." Unifies `email.js`'s `spawnSync({input})` with `backup.js`'s env+timeout;
  the only genuinely new code is the `when` gate + config parse. (`checks.js:exec()` is the
  wrong primitive — `execFile`, no stdin.) The extraction also gave `sendEmail` the handoff
  exit-status reporting (`{transport, ok}`) that `on-primary-failure` reads.

**Surface.** `fallback: { command, args?, when?, timeoutMs? }`. Body → child stdin; subject
→ `PULSELOG_SUBJECT`. A malformed block (no `command`) fails the run **loud** (exit 1), like
other bad config; runtime spawn failures are best-effort.

**Refusals (unchanged from proposal).** No webhook v1; no pluggable-transport framework /
retry queue / multi-fallback fan-out; not a native maillog/bounce check-type (stays in
`examples/`); no cross-run dedup (stateless by design). `sendmail` exit 0 = *queued* not
*delivered* — the accepted limitation `when:"always"` + the off-box recipe (below) cover.

**Companion (docs).** `examples/mail-delivery-check.sh` — an off-box "is my VPS still
delivering?" `command` check, to be ported faithfully from the real running script
`addypin/ops/homeserver/addypin-mail-check.sh` (latest-outcome logic, `|| true` → green on
SSH-down, parameterized host/user/key/recipient, Postfix-specific maillog matcher noted).

**Validation.** POC-first (riskiest assumption: best-effort stdin sink that never throws /
never changes exit code — proven with an A/B control + producible negatives). Built in two
modules (spawnDelivery + `sendEmail` handoff reporting; then the wrapper across all three
sites). 67/67 tests green; end-to-end through the real CLI with a silent-on-green negative
control. Build ≥1 incident + generic to every deployment → earns its place.
